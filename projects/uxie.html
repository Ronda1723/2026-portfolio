CASE STUDY · 01 2024 — NOW

Uxie

Voice-first writing for Mac & Windows.

Hold a key. Talk. Let go. Clean text appears wherever your cursor is. Uxie is a small, signed desktop app that turns dictation from a separate workflow into a keystroke — built around a single obsession: it has to feel instant.

~300msperceived latency, p95
Mac + Winshipped same day
v1.0.12+weekly releases

The insight

We watched a dozen people use dictation tools — built-in macOS dictation, Whisper Flow, Wispr Flow, native Windows speech. The same shape of complaint kept coming up: "It's faster than typing, but it doesn't feel like typing." The hand had to stop, the cursor had to move, a separate window had to open. The tool wasn't a key on the keyboard — it was a destination.

The opportunity wasn't in transcription quality (Whisper-class models had already solved that). It was in fitting into the writing flow people already had. Make speech a modifier key, not an app.

Who it's for

Three concentric circles, smallest to largest:

  • Heavy writers — engineers in Slack and Linear, founders in Notion, designers in Figma comments. People typing 8+ hours a day who would happily trade a keystroke for a sentence.
  • Multilingual professionals — speaking is faster than typing in any language; for non-native English typists the speed-up is 3–5×.
  • Anyone with hands that hurt — a quietly large user base. Voice-as-modifier is meaningfully different from "open the dictation app."

The early adopter cohort was almost entirely heavy writers. The other two circles came naturally once the product felt fast enough that the floor of "good enough" was no longer in question.

The product, in one sentence

A signed, native desktop app that lives in the menubar (Mac) / system tray (Windows), listens only while you hold a global hotkey, streams audio to a hardened backend, and dispatches the resulting text into whatever app your cursor is in.

The interaction model

The whole product is one gesture: press → speak → release. There's no "start," no "stop," no commit button.

Three small design decisions made this feel right:

  • Dispatch on release, not on silence. Most tools wait for a pause to decide you're done. Pauses are unreliable (people breathe, hesitate, think). Tying dispatch to the modifier key release gives the user a physical commit gesture — same as letting go of a paint stroke.
  • Hold-to-talk by default, toggle-to-talk as an option. Hold is the right default for short utterances (the 80% case). Long monologues — meetings, voice notes — get a toggle mode behind a setting.
  • The hotkey is rebindable per app. One shortcut in Slack, another in your IDE. The OS already has app-scoped state; we just expose it.

The states (designer notes)

Most desktop apps over-design state feedback because they have screen real estate to fill. Uxie has none — it lives as a 22px menubar icon. Every state had to be expressible in that single dot.

  • Idle — outlined circle. Quiet. Forgettable.
  • Listening — fills with the system accent color, with a subtle 2px ring that pulses softly with input amplitude. No bouncing waveform; that reads as "show-off."
  • Processing — same fill, but the ring spins. ~400ms max — if it spins longer, something's wrong.
  • Dispatched — flashes to checkmark, fades back to idle over 220ms. Short enough not to be noise, long enough to confirm.
  • Error — fills red, then back to idle. Errors are clickable to see what happened; no modal interrupts the writing flow.

Audio cues exist but are off by default. Most heavy writers use headphones for music; a notification chirp on every utterance gets old in an hour.

Latency: the actual product

Transcription quality plateaued in 2024. What didn't plateau was perceived latency. Everything important about Uxie is downstream of one number: how long between key release and visible text.

The combo that landed v1.0.12 at sub-300ms p95:

  • Streaming Deepgram Nova-3. Returns interim transcripts as you speak. We discard interim cuts and only commit on the final boundary — but the model has already done most of the work by the time you release.
  • Dispatch on release. Detailed above — turning a "wait for silence" pause into a physical gesture saves 200–400ms perceptually, even if wall-clock is identical.
  • Key cache. Hotkey state, last clipboard, last cursor position — pre-warmed in memory so dispatch doesn't query the OS. Saves ~20–60ms but, more importantly, removes jitter.
  • Server-side keys. All Deepgram and LLM calls go through a Railway backend. No client-bundled keys (security), and the round-trip stays inside one trusted hop with auth via Deepgram's Token scheme.

Cross-platform parity, not "Mac-first"

The temptation with consumer-grade software is to ship Mac first, Windows "when we get to it." We ruled this out early. Half our heaviest target users are on Windows (engineers, founders in fintech, anyone whose company issued a Dell). A six-month parity gap would be a six-month gap in trust.

The design constraints that fell out of "Mac and Windows on the same day":

  • One binary lineage. Built on Tauri so the visual layer is web-rendered identically on both, and the native shell handles OS-specific bits (hotkey registration, signing, autostart).
  • Same version, same tag. Releases ship as v1.0.12 on both. Internal rule: a release isn't a release until both signed artifacts (DMG + EXE) are attached to the same GitHub tag.
  • Visual language stays platform-honest. Mac uses SF Pro, Windows uses Segoe UI Variable. Same layout, same metrics, native typography. No "Mac UI on Windows" or vice versa — that always looks like a beta.

What we measure

One north star, four supporting metrics:

  • North star — utterances per active user per day. If people are reaching for the hotkey, the product works. Vanity DAU doesn't tell us this; sessions can be a single forgotten install.
  • p95 perceived latency. Capped at 350ms. Anything over goes to a regression triage queue.
  • Dispatch success rate. % of utterances where text actually landed in the destination app without user retry.
  • D7 retention by activation cohort. Users who hit ≥5 utterances on day 1 vs. those who didn't — the gap tells us how good the first-run flow is.
  • Crash-free sessions. Boring, important. The bar is 99.5%.

What's next

  • Word-level edit gestures. "Scratch that," "fix the last sentence." Voice as a small command surface, not just dictation.
  • Per-language model swapping. Heavy bilingual users want to switch between English and Hindi/Mandarin/etc. without changing settings — detect-and-swap.
  • Offline fallback. A small on-device model for short utterances when the network is bad. Same UX, lower-quality result, but the gesture still works.
  • Selection-aware rewrites. Select text → hotkey → "make this less formal." The lift is small; the value is large.

The constraint stays the same: anything that doesn't preserve the "press, speak, done" rhythm doesn't ship.
