CASE STUDY · 01 2024 — NOW

Uxie

Voice-first AI agent for macOS · Mac-native · Single-developer indie

Uxie is a Mac-native voice agent that lives at the OS layer. Users hold the fn key from any app and speak — Uxie dictates into the focused window, runs agent commands across their connected services (Gmail, Calendar, Drive, Slack), records meetings with both their mic and the other participants' audio, and delivers a synthesized morning brief at a user-chosen time. The product is voice-first, ambient, OS-deep — built for the user category that increasingly resents typing into a chat box to talk to AI. Today: shipping v1.5.1 to a private beta of trusted users via Google OAuth Testing mode. Production launch gated on Google OAuth verification (~6 weeks elapsed) or Composio adoption (instant).

The Problem

The market has fragmented into point-solution AI tools, each owning one narrow surface:

ToolWhat it doesWhat it misses
GranolaMeeting transcription + summaryNo voice surface, no cross-tool agency
Wispr Flow / SuperwhisperVoice typingNo memory, no tools, no context
ChatGPT desktopChat-bot Q&ASandboxed in a chat window, no OS depth
CursorIDE coding agentOne context, dev-only
Multi-On / OperatorBrowser automationNo local context, fragile

None of these talk to each other, share memory, or compound. The user has to context-switch between five tools to get one cohesive AI assistant experience.

The actual gap: a personal AI that knows the user well enough to be useful and is trusted enough to act on their behalf — across all their tools, with persistent context, voice-first, OS-native. Whoever builds that wins the consumer market for the next decade.

Specific user pain points Uxie addresses

  1. Typing is the wrong primary input for AI. Speaking is 3-5x faster, frees hands, doesn't break flow. But every existing AI tool defaults to a text box.
  2. Meeting notes are still manual or require a bot to join the call. Granola solved "bot-free" but users want their meetings tied into a larger AI workspace, not a standalone notepad.
  3. AI agents are reactive — they wait to be asked. Users want the AI to wake them up with a morning brief, summarize their day at 6pm, surface that one email they forgot to reply to.
  4. "Approve before action" is a UX afterthought in most tools. Uxie treats it as the central trust mechanism.
  5. Existing voice tools have no context. They transcribe but can't say "reply to the email from Sarah about pricing" because they don't know what that thread says.

Why Now

Three macros are converging:

  1. macOS ScreenCaptureKit + AVCaptureSession (2023+) made system-audio capture for meeting transcription a privacy-respecting, no-driver-required reality. Granola proved the market for this.
  2. STT quality / latency crossed the "good enough to actually transcribe a meeting" threshold (Deepgram Nova-3, AssemblyAI Universal-2).
  3. Tool-use LLMs (GPT-4o, Claude 3.5+) made it practical to chain "read this email, draft a reply, ship it for approval" across multiple connectors in a single agent turn.

The combination of these three was not available 18 months ago. The window to ship a voice-first, OS-native, agentic Mac app is now.

Direction & Strategy

The thesis

Context is the product, not the model.

Models are fungible (today GPT-4o, tomorrow Sonnet 5, next year a local Llama). What's not fungible is the corpus of the user's data the agent has indexed. The moat is built one transcript, one dictation, one captured meeting at a time. Memory is the only thing that compounds. Build it early; let model quality be someone else's problem.

Five concrete bets

  1. Voice-first habit > chatbox UX. The hotkey is the wedge. Once a user dictates emails for a week, they don't go back.
  2. OS-level depth > browser sandbox. Capture system audio, watch frontmost-app changes, type into any window via Accessibility — the things a Chrome extension or web app fundamentally cannot do.
  3. Ambient > reactive. Scheduled briefings, auto-detected meeting recording, proactive notifications. The product works on the user's behalf while they sleep.
  4. Trust through transparency. Every destructive action (gmail_send, slack_send_message, calendar_create_event) gates on an explicit user approval card — with editable params inline.
  5. Patience > velocity. Granola took 18 months to nail one feature. Uxie is doing four in parallel, but each one is held to "actually-works" quality before shipping. The hotfix-and-bisect cycle is part of the process, not a sign of failure.

The Solution

Uxie is a single macOS app that bundles five tightly-integrated capabilities. Each capability is one tab in the left rail, plus an always-listening hotkey:

┌──────────────────────────────────────────────────────────────┐
│  Sidebar tabs (left rail):                                   │
│    ⌂  Home          — landing / status                       │
│    ⚡  Tasks         — background agent tasks                 │
│    ☀  Briefings     — morning brief / end-of-day / weekly    │
│    📝 Meetings      — recorded meetings + transcripts        │
│    📖 Dictionary    — spoken-form → text replacements        │
│    ✎  Snippets      — text expansion triggers                │
│                                                              │
│  Always-on:                                                  │
│    fn-hotkey         — dictation + voice commands            │
│    Auto-detect       — watches for Zoom/Slack/Meet windows   │
│    Cron worker       — fires scheduled briefings on time     │
└──────────────────────────────────────────────────────────────┘

Capability 1 — Voice surface (the wedge)

The user holds fn from any macOS app, speaks, releases. Uxie:

  1. Captures bundle ID of the focused app at the moment of press
  2. Reads any highlighted text via the Accessibility API
  3. Streams mic audio to Deepgram via Railway-issued ephemeral key
  4. Runs grammar correction (Groq Llama 3.3 70B for dictation; GPT-4o for commands)
  5. Types the corrected text back into the focused app via synthetic CGEvent keypresses

Behavioral split:

  • Dictation (no command keyword) → grammar-clean and type into the focused window
  • Command (e.g., "send Sarah an email saying the proposal looks great") → agent loop with tool-calling, ends with an approval card before any destructive action
  • Transform (text selected + transform keyword like "polish this") → LLM rewrites the selection in place

Capability 2 — Meetings (Granola-class)

The hard part of voice-first AI. Built around a Swift sub-bundle UxieAudioTap.app inside the main Uxie.app.

Pipeline:

AVCaptureSession (mic, hw rate)  ─┐
                                  │ resample → 16 kHz mono Float32
ScreenCaptureKit  (system audio) ─┘
                                  ▼
                          mix sample-by-sample (mic = master clock)
                          clip → Int16 LE PCM
                                  ▼
                          stdout (raw bytes, no header)
                                  ▼
                          Python audio_meeting.py
                          ─ pipe to Deepgram WS (long-form, diarize)
                          ─ tee to local WAV at ~/miniflow/meetings_audio/<id>.wav
                                  ▼
                          Deepgram returns interim + final transcripts
                                  ▼
                          appended to SQLite at ~/miniflow/meetings.db
                          + broadcast "meeting:transcript-update" to UI
                                  ▼
                          On "Structure this meeting" click:
                          → Railway /llm/structure-meeting
                          → GPT-4o with Granola-style prompt
                          → returns Markdown: TL;DR / Decisions / Action Items

Auto-detection runs as a second mode of the same Swift binary (--watch). Polls SCShareableContent every 5 seconds. Matches window bundle IDs + titles against a curated list of meeting apps (Slack huddle, Zoom Meeting, Teams Meeting, Webex, Google Meet in any browser). When a match appears, fires a native macOS notification with Record / Skip action buttons.

Capability 3 — Background tasks ("Tasks" tab)

User types a prompt → it runs detached on Railway → polling UI shows live progress → notification on completion.

Architecture:

  • POST /tasks/create returns task_id immediately, fires asyncio.create_task in the background
  • Background loop runs OpenAI tool-calling with the user's connected tools
  • Read-only tools (gmail_search, calendar_list_events, drive_search, slack_search) run freely
  • Destructive tools (gmail_send, slack_send_message, calendar_create_event) park on an asyncio.Event until the user clicks Approve in the inline approval card
  • Each event in the agent loop (thinking, tool_call, tool_result, approval_needed, final_text) is appended to a task_events table; the Mac client polls /tasks/{id} every 2s while running
  • Parallel tool calls within a turn run via asyncio.gather — "summarize my unread Gmail AND check my calendar" fires two workers concurrently

Capability 4 — Scheduled briefings ("Briefings" tab)

Recurring workflows that fire at a user-chosen local time. Three templates ship today:

TemplateDefault timeWhat it does
Morning Brief08:00Parallel fans out to Gmail + Calendar + Slack, synthesizes today's schedule + inbox highlights + heads-up
End-of-Day Recap18:00Past-day sent emails + meetings attended → "What you shipped today, what's still open"
Weekly Digest17:00 FridayPast-week sent emails + meetings → "Your week in three bullets, standout meetings, open threads"

Backend scheduled_tasks table + cron worker on Railway's lifespan polls every 60 seconds. Each template is a generator function in TEMPLATE_REGISTRY — adding a fourth template = one entry in the dict + one generator function.

Delivery is dual-channel: macOS notification (banner with the brief's TL;DR) + email via Resend (full Markdown rendered as HTML). User toggles either independently in the time-picker card.

Capability 5 — Referral system

Every user gets a unique 8-character referral code at signup. Sharing the link uxie.ai/r/<code> and having a friend activate gives both parties +30 days of Uxie Pro. Stats shown in Settings → Invite tab: friends joined, days earned, current Pro balance, redeem-a-code input for the inverse path.

Design Principles

1. Mac-native everything

No web app pretending to be a desktop app. Uxie uses real macOS chrome, real notifications, real keychain storage for JWTs, real Accessibility APIs for synthetic typing, real NSWorkspace.frontmostApplication queries. Users feel the platform.

2. Voice as primary input, visual as fallback

Voice for capture (faster than typing). Visual for approval (faster than listening to "the email reads...do you approve?"). Notch / overlay UI for status. Audio TTS for hands-busy output ("driving home — what happened today?"). The right mode for the moment.

3. Warm, low-stakes visual language

Background: #F3F3F1 (warm cream). Primary text #1a1a1a (near-black, not pure). Muted text #666-888. Action accents:

  • Primary actions: filled near-black background (#1a1a1a bg, white text), 6-8px border radius
  • Secondary actions: outlined 1px solid #ccc with transparent background
  • Destructive: outlined 1px solid #d44a4a red

Typography stack: system default (-apple-system, BlinkMacSystemFont, "Segoe UI"). 13-15px body, 10-11px section labels in uppercase letterSpacing 0.05 for the "lighting up" effect. Monospace for code/IDs (ui-monospace).

Spacing on an 8-point grid. Most cards have 16px internal padding, 12-16px gap between siblings, 24px between sections. The product has a deliberately spare density — not Notion-dense, not Linear-tight. Closer to Granola's calm.

4. Status pills as the universal UX vocabulary

Every entity (meeting, task, schedule, OAuth connection) has a status pill. Same shape, same height, same letter-spacing. The user learns the visual grammar once: "amber = needs me, green = healthy, gray = idle, red = broken". This is how a complex agent product stays cognitively cheap.

5. Approval cards as the central trust mechanism

Every destructive action (sending email, posting Slack, creating calendar event) gates on an approval card with:

  • A clear human-readable summary at the top
  • A collapsible inline editor where users can fix typos in recipient / subject / body before approving
  • Two buttons: Approve (filled), Decline (outlined)
  • 5-minute timeout → auto-decline

This is more important than the chat history. Trust is earned per approval, not assumed.

6. Never claim functionality you don't have

If a feature fails (Deepgram 401, Composio rate limit, OAuth scope missing), the error message reaches the user with an actionable next step — not a stack trace, not a vague "something went wrong". Backend translates raw HTTP errors into instructions ("Your Google connection has expired — Settings → Connectors → reconnect"). The LLM passes these through verbatim.

Why Railway sits in the middle: users never see provider API keys. The backend mints short-lived Deepgram tokens, proxies LLM calls through Groq/OpenAI, holds OAuth refresh tokens server-side so the .app bundle has zero secrets. Per-user rate limits + per-tier quotas live in limits.py.

Why a Swift sub-bundle for audio: raw CLI binaries fail TCC silently on macOS — no Info.plist for the system to read usage descriptions from. Wrapping the Swift binary in a tiny UxieAudioTap.app inside Resources/ gives TCC something to attach permissions to. This was a multi-iteration discovery (AVAudioEngine vs AVCaptureSession, missing Info.plist, CLI-mode quirks) — documented in detail in STATE.md.

Privacy Stance

Local-first by default. Granola's "listens, writes, forgets" architecture but with an explicit opt-in for users who want centralized recordings:

  • Default: meeting transcripts stay in ~/miniflow/meetings.db on the Mac. Audio is retained locally as WAV at ~/miniflow/meetings_audio/<id>.wav for replay.
  • Opt-in toggle in Settings → Account → "Share meetings with admin dashboard". When ON, audio + first 2000 chars of transcript upload to Cloudflare R2 + Railway Postgres for admin review.
  • Voice commands (short hotkey utterances) discarded after transcription unless admin debug mode is on.
  • OAuth tokens held server-side on Railway (Postgres), JWT on the Mac in macOS Keychain. The .dmg ships with zero credentials.
  • PRIVACY.md + TERMS.md drafted in the repo, ready to host at uxie.ai/privacy and uxie.ai/terms for OAuth verification.

Roadmap

Near-term (next 4-8 weeks)

  • Submit for Google OAuth verification (or migrate sensitive scope handling to Composio as a faster path)
  • Stand up landing page at uxie.ai
  • Wire Stripe for Pro tier
  • Expand keyword vocabulary boost (currently hardcoded list) to pull from user's calendar attendees + address book
  • Possibly switch STT provider to AssemblyAI for the latency win in Asia-region users

Mid-term (3-6 months)

  • Vector memory layer — every transcript + dictation + meeting indexed; "what did Mike say about the contract?" becomes a one-sentence voice query
  • Proactive whispers — "Sarah replied to your email — want to draft a response?" — surfaced as macOS notifications
  • More connectors via Composio (Notion, Linear, Asana) without rebuilding OAuth per provider
  • iOS companion app — read meetings, run tasks from phone

Long-term

  • Personal voice / style learning — Uxie writes emails in the user's voice, not generic LLM voice
  • Workflow gallery — user-shareable scheduled workflows beyond the three baked-in templates
  • Team features — shared meeting library, team transcripts (when single-player retention is strong)
